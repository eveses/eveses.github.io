## 1. AWS RDS 개요
<h3>RDS란 무엇인가?</h3>

> RDS(Relational Database Service)는 AWS에서 제공하는 관리형 데이터베이스 서비스이다. 하드웨어 프로비저닝, 데이터베이스 설정, 패치 및 백업과 같은 운영 작업을 AWS가 자동화해주므로, 사용자는 애플리케이션 개발에만 집중할 수 있다.

<h3>지원하는 데이터베이스</h3>

- Postgres
- MySQL
- MariaDB
- Oracle
- Microsoft SQL Server
- Aurora

<h3>RDS Custom</h3>

> OS와 DB에 대한 제어 권한과 관리형 서비스의 이점을 동시에 가질 수 있는 서비스로 Oracle과 Microsoft SQL Server만 가능.

- SSH나 RDP로 OS 접속 가능
- OS 레벨의 커스터마이징이나 타사 소프트웨어 설치 가능

## 2. RDS 핵심 아키텍처 및 주요 기능

<h3>Multi-AZ (고가용성 및 자동 장애 조치)</h3>

- 동기식 복제 SYNC
- 다른 AZ에 대기용 인스턴스 생성
- 대기용 인스턴스는 접속 불가
- 생성과정: 메인 DB 스냅샷 -> 대기용 DB 생성 -> 동기화

<h3>Read Replicas (읽기 성능 확장 및 DR)</h3>

- Cross-Region, 비동기식 ASYNC
- DB애서 수정/삭제, Read Replicas는 읽기만
- 특정 복제본을 독립적인 DB로 승격 가능
- 기본적으로 다른 AZ로 데이터가 이동하면 비용이 발생하지만 Read Replicas같은 managed service는 예외 / cross-region은 비용 발생

<h3>Storage Auto Scaling</h3>

> 예상치 못한 데이터 증가에 유연하게 대처하기 위한 기능

- 최대 스토리지 임계값을 설정
- 워크로드 예측이 어려운 애플리케이션에 유용

<h3>RDS Proxy (Connection Pooling)</h3>

> 애플리케이션 - 프록시 - RDS 구조

- 장애조치 시간 감소
- 람다 함수와 같이 사용할때 효율적
- 퍼블릭 엑세스 불가
- DB에 대해 IAM 접근 강제
- DB 자격증명을 Secrets Manager에 저장

## 3. Amazon Aurora
<h3>Aurora 아키텍처 특징 (컴퓨팅/스토리지 분리)</h3>

- **Shared Storage Volume:** 컴퓨팅 노드(DB 인스턴스)와 스토리지 층이 분리되어 있음
- **3 AZ, 6 Copies:** 데이터는 3개의 가용 영역(AZ)에 걸쳐 6개의 복사본으로 저장됨 (높은 내구성)
- **Quorum:** 쓰기 작업은 6개 중 4개, 읽기 작업은 3개의 노드 동의 필요
- 스토리지 자동 확장 (10GB ~ 128TB)
- **Aurora Replica:** 일반 RDS Read Replica와 달리 스토리지를 공유하므로 복제 지연(Lag)이 거의 없음 (ms 단위)

<h3>Aurora Endpoints 및 Custom Endpoint</h3>

> Read Replicas 중 성능이 좋은 Replica가 따로 있다면 그것과 Custom Endpoint를 설정하여 복잡한 쿼리는 해당 Replica로 보내는 식으로 사용

- **Cluster Endpoint (Writer):** 쓰기 작업이 가능한 Primary 인스턴스로 연결
- **Reader Endpoint:** 연결된 모든 Read Replica로 로드 밸런싱
- **Custom Endpoint:** 특정 인스턴스 그룹(예: 고성능 인스턴스)을 묶어 별도의 엔드포인트 생성. 분석용 복잡한 쿼리를 처리하는 Replica 그룹에만 트래픽을 보낼 때 유용

<h3>Aurora Serverless (Auto Scaling)</h3>

- 트래픽이 들어오는 만큼 알아서 DB 사이즈 조절 (ACU 단위)
- 인스턴스 타입 선택 불가, 자동 스케일링
- 트래픽 예측이 어려울 경우 사용

<h3>Global Aurora (Global Database)</h3>

- 데이터베이스 복구에 사용
- 하나의 Primary Region, 최대 10개에 Secondary Regions
- RTO < 1 minutes

<h3>Babelfish for Aurora Postgres</h3>

> 중간에서 Microsoft SQL 명령어로 PostgreSqL 데이터베이스를 사용 가능하게 해주는 번역 기능.

- 애플리케이션 코드를 수정하지 않고도, 기존 SQL Server용 쿼리(T-SQL)를 Aurora PostgreSQL이 이해하고 처리하도록 해주는 **통역(Translation) 레이어**
- SQL Server 라이선스 비용 절감 목적 (Migration 가속화)

<h3>Aurora Cloning</h3>

- **Copy-on-Write 프로토콜:** 데이터를 물리적으로 복사하지 않고 포인터만 생성하므로 **생성 속도가 매우 빠름 (즉시)**
- 원본 DB에 영향을 주지 않고 **Staging(테스트) 환경 구성**이나 **데이터 분석**용 DB를 만들 때 유용

## 4. 백업 및 복원

<h3>Automated Backup (자동 백업 및 보존 기간)</h3>

- 비활성화 할 수 있고, 백업본을 1~35일간 유지할 수 있다.

<h3>Manual DB Snapshot</h3>

- 보존 기간을 마음대로 설정할 수 있다.
- 이를 통해 DB를 오랫동안 사용하지 않을 경우 스냅샷 찍은 후 기존 DB는 삭제하는 식으로 비용 절감 가능

<h3>Restore Options (PITR 등)</h3>

- Create new DB: 기존 DB를 덮어쓰는 것이 아니라, 항상 새로운 주소(Endpoint)를 가진 새 DB가 생성됨
- Restore from S3: On-premise(로컬) 데이터베이스 백업(MySQL 등)을 S3에 올린 후 RDS로 복원 가능

## 5. 보안 및 모니터링 (Security & Monitoring)

<h3>데이터 암호화 (At rest / In transit)</h3>

- **At-rest (저장 데이터):** AWS KMS(Key Management Service)를 사용하여 볼륨 전체 암호화 (생성 시에만 설정 가능, 추후 변경 불가)
- **In-transit (전송 데이터):** 애플리케이션과 DB 간 SSL/TLS 인증서 사용
- **스냅샷 암호화:** 원본 DB가 암호화되어 있으면 스냅샷과 복원된 DB도 자동 암호화

<h3>IAM Database Authentication</h3>

- DB 자체 비밀번호 대신 **IAM 역할(Role)과 토큰**을 사용하여 로그인
- **장점:** 비밀번호 관리 불필요, EC2/Lambda와 연동 시 보안성 및 편의성 증대 (MySQL, PostgreSQL 지원)

## 6. 연관 서비스

<h3>ElastiCache (In-memory Caching)</h3>

- 자주 찾는 데이터는 캐시에 저장
- Redis VS Memcached

- **Redis:**

    - 데이터 영구 저장 가능, 고가용성(Multi-AZ), 백업/복원 지원
    - **게임 리더보드 (Sorted Sets)**, 지리 공간 데이터, 복잡한 데이터 구조

- **Memcached:**
    - 단순 객체 캐싱, 멀티스레드 지원 (수평 확장 용이)
    - 데이터 영구 저장 불가 (휘발성)
