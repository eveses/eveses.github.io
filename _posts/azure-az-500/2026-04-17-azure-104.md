# [계정 및 권한 관리]

## Microsoft Entra ID

![](/assets/images/azure/az-500/스크린샷%202026-04-18%20오전%2010.52.55.png)

### Azure 기본 구조  
> Azure는 크게 하나의 Entra ID(테넌트) 안에서 여러 Management Group, Subscription을 배치하여 계층 구조를 만들 수 있음

#### Azure Account (Entra ID)  
- Azure의 최상위 논리적 경계인 테넌트
- 조직의 ID(사용자, 그룹, 서비스)가 저장되는 전용 디렉터리

#### Management Group
- 여러가지 Subscriptions를 관리하기 위한 컨테이너
- AWS의 OU와 유사한 개념으로 최대 6단계까지 계층 구조
- Azure policy나 RBAC을 Management Group수준에 할당하면 그 아래 속한 모든 Subscriptions와 Resources에 자동 적용

#### Subscription
- Azure 리소스를 사용하는 기본 단위
- 각 Subscription은 리소스의 한도와 비용 청구 경계

#### Resource Groups
- Azure 리소스를 담는 논리적인 컨테이너
- 동일한 서비스, 프로젝트에 사용하는 리소스를 하나로 묶어 관리
- 리소스 그룹 단위로 권한 부여 가능

#### Resources
- Azure에서 생성하는 최소 단위의 서비스 객체
- VM, Vnet, DB 등

### Entra ID

> Entra ID는 Microsoft 클라우드 기반 ID 및 액세스 관리 서비스.

![](/assets/images/azure/az-500/스크린샷%202026-04-18%20오후%201.15.04.png)

#### Entra ID 역할
- **인증**: 액세스 권한을 부여하기 전 ID 확인
- **SSO**: 한 ID가 여러 애플리케이션에 액세스 가능
- **애플리케이션 관리**: 애플리케이션 프록시, SaaS 앱 통합 및 내 앱 포털과 같은 기능을 통해 클라우드 및 온프레미스 앱 관리
- **디바이스 관리**: Microsoft Intune과 같은 도구를 통해 디바이스 등록 및 관리

#### Azure 인증 방법
> 패스워드 로그인, MFA, 암호없는 로그인 가능

![](/assets/images/azure/az-500/스크린샷%202026-04-18%20오후%201.23.41.png)

## Azure RBAC

### 보안 주체
- **사용자 및 그룹**: 실제 사용자나 조직 단위
- **서비스 주체**: 외부 환경의 애플리케이션이나 자동화 파이프라인 등이 사용하는 자격 증명
- **관리형 ID**: Azure 리소스 (VM, App Service)자체가 가지는 독립적인 ID로 AWS의 인스턴스 프로파일과 유사

### 역할 정의
- Azure Role은 AWS의 정책과 유사 개념
- 위 보안 주체에 부여 가능
- Built-in Role: Owner(소유자), Contributor(기여자), Reader(읽기) 등 수백개의 사전 정의된 Role 존재
- Custom Role: 최소 권한 원칙을 엄격히 지켜야 할 경우 역할 커스텀 가능 (json)

### 범위
- 할당한 역할이 적용되는 범위
- 상위 수준에 부여한 역할이 하위로 자동 상속
- 관리그룹 > 구독 > 리소스 그룹 > 개별 리소스

# [네트워킹 서비스]

## Virtual Network

### Vnet 기본 구조
- **Vnet**: 리전 단위의 사설 네트워크로 하나의 Vnet 내에서 사설 IP 대역 정의
- **Subnet**: Vnet을 더 작은 단위로 쪼갠 것
    - AWS와 다르게 서브넷이 특정 가용영역에 종속되지 않고 리전 전체를 커버
    - 하나의 서브넷 안에 여러 가용영역에 있는 VM 배치 가능
- **IP 주소**
    - Private IP: 리소스간 내부 통신용
    - Public IP: 인터넷 연결용 (Azure는 기본적으로 외부 인터넷으로의 아웃바운드 연결은 허용되었으나 25년 말 차단으로 변경?)
        - 표준(유료) SKU는 정적 할당만 가능(고정) 기본(무료) SKU는 동적 할당만 가능

### 네트워크 보안 제어
- **Network Security Group (NSG)**: L3/L4 상태저장 방화벽
    - 서브넷이나 개별 NIC에 적용 (AWS처럼 SG, NACL 나눠져 있지 않고 하나로 통일)
    - 서비스 태그: 특정 Azure 서비스의 IP대역을 이름으로 묶어둔 태그
- **Application Security Group (ASG)**: IP 주소가 아닌 애플리케이션 역할을 기준으로 설정
    - ex) web-server 태그를 가진 모든 VM은 DB-server 태그를 가진 3306포트로만 접근 가능 (AWS의 보안그룹 참조와 유사)

## 연결
- **Vnet Peering**: 두 Vnet을 백본 네트워크를 통해 연결
    - Hub and Spoke: 중앙에 공유 서비스를 두고 여러 Vnet 연결
- **Azure Private Link/Private Endpoint**: PaaS 서비스를 외부인터넷이 아닌 내부 사설 IP로 접근

## Load Balancer
> Azure Load Balancer는 4계층에서 작동하며 들어오는 네트워크 트래픽을 백엔드 리소스(VM)로 균등하게 분산해주는 서비스로 AWS NLB와 유사  
트래픽은 5튜플을 사용하여 관리 (원본IP, 원본포트, 대상IP, 대상포트, 프로토콜)

![](/assets/images/azure/az-500/스크린샷%202026-04-18%20오후%202.36.55.png)

### 구성 요소
- **프런트 앤드 IP**: 외부 또는 내부에서 트래픽이 들어오는 관문으로 공용IP/사설IP 할당 가능
- **백엔드 풀**: 트래픽을 전달받을 리소스 집합으로 주로 가상머신이나 가상머신 확장 집합(VMSS)
- H**ealth Probes**: 백엔드 풀의 리소스가 정상인지 주기적으로 확인하고 응답이 없는 VM은 전달 대상에서 제외
- **로드 밸런싱 규칙**: 프런트 앤드 IP로 들어온 특정 포트의 트래픽을 백앤드 풀의 어떤 포트로 전달할지 정하는 규칙
- **인바운드 NAT 규칙**: 특정 포트로 들어온 트래픽을 백앤드 풀의 특정 VM에 1:1로 전달
    - ex) 50001포트로 접속하면 1번 VM의 3306 포트로 전달

## Application Gateway
> Azure Application Gateway는 클라이언트 애플리케이션이 웹 서버 풀에서 호스트되는 웹 앱에 보내는 요청을 관리  
웹서버 풀은 Azure VM, VMSS, App Service 및 온프레미스 서버 등.

![](/assets/images/azure/az-500/스크린샷%202026-04-18%20오후%203.29.21.png)

### 구성 요소
- **프런트 앤드 IP**: 외부 또는 내부에서 트래픽이 들어오는 관문으로 공용IP/사설IP 할당 가능
- **리스너**: 특정 포트(80/443), 프로토콜, IP/호스트 이름으로 들어오는 트래픽을 대기하고 수락 (여기서 TLS 인증서 처리(복호화))
- **라우팅 규칙**: 리스너가 받은 트래픽을 어느 백앤드 풀로 보낼지 결정
- **백앤드 풀**: 실제 서비스가 구동되는 대상

### 라우팅 패턴
- **경로 기반 라우팅**: URL뒷부분 (디렉터리)를 보고 라우팅
- **다중 사이트 라우팅**: URL앞부분 (호스트 도메인)을 보고 라우팅

### 추가 기능
- **WAF**: 리스너에 트래픽이 닿기 전 규칙 기반으로 웹 해킹 공격 차단 (sku에 따라 적용 가능)
- **Health Probes**: 백엔드 서버가 200~399 정상 코드를 반환하는지 지속적으로 검사하고 장애가 난 서버로는 트래픽을 전송하지 않음
- **오트스케일링**: 트래픽 증감에 따라 AGW 인스턴스가 자동으로 스케일 업/다운

## Azure Front Door
> 사용자가 접속할 때 트래픽이 특정 리전까지 길게 인터넷망을 타고 오지 않고 사용자와 가까운 엣지 POP가 트래픽을 먼저 수신 후 고속 백본망을 타고 서버로 전송

![](/assets/images/azure/az-500/스크린샷%202026-04-18%20오후%203.51.18.png)

### 주요 기능
- **부하 분산**: 단순히 순서대로 트래픽을 분산하지 않고, HTTP 헤더, URL 경로, 정규식 등을 분석하여 트래픽 분산
- **상태 검사**: 엣지 POP들이 주기적으로 백엔드 서버 상태 검사. 특정 리전 서버 다운 시 다른 리전으로 우회
- **리다이렉션**: http로 접속해도 강제로 https로 접속하도록 유도

# [컴퓨팅 서비스]

## Virtual Machines
> VM은 Azure의 가장 대표적인 IaaS 리소스로, 클라우드 환경에서의 가상 서버.

### VM Sizing (인스턴스 패밀리)
- **General Purpose (D시리즈)**: 가장 범용적인 사양으로 CPU와 메모리 비율이 균형 잡혀 있어 웹 서버나 소규모 DB에 적합
- **Compute Optimized (F시리즈)**:CPU 성능이 뛰어나 배치 처리나 과학적 계산 등 연산 위주의 작업에 사용
- **Memory Optimized (E.M시리즈)**: 대용량 메모리를 탑재하여 SAP HANA, Redis 등 인메모리 DB나 대규모 데이터 분석에 필수
- **Storage Optimized (L시리즈)**: 로컬 NVMe 스토리지를 탑재하여 매우 높은 디스크 IOPS를 제공. NoSQL DB 등에 유리
- **GPU Optimized (N시리즈)**: 그래픽 렌더링이나 딥러닝 모델 학습을 위한 NVIDIA GPU를 제공

### Managed Disks
> VM의 데이터는 가상 하드디스크 형태의 Managed Disk에 저장되고 성능 수준에 따라 4가지 등급으로 나뉨

- Standard HDD: 백업이나 드물게 액세스 하는 데이터용 (가장 저렴)
- Standard SSD: 가벼운 개발/테스트용
- Premium SSD / SSD v2: 프로덕션 환경의 실질적인 표준, 지연시간이 낮고 일관된 성능 제공
- Ultra Disk: 최고 사양, 실행 중에도 IOPS와 처리량을 동적으로 조정 가능

### 요금 모델
- **종량제 (Pay as you go)**: 약정 없이 켜둔 시간 만큼 비용 지불
- **예약 인스턴스**: 특정 리전의 특정 시리즈를 1/3년 동안 약정. 종량제 대비 최대 72프로 할인
- **컴퓨팅 절감 플랜**: 특정 서버를 지정하는 RI와 달리, 1/3년동안 Azure 전체에서 시간당 특정 달러 어치의 컴퓨팅 사용을 약정. 최대 65프로 할인
- **스팟 가상머신**: Azure 데이터센터에 남는 자원을 빌려 사용. 최대 90프로 할인
- **전용 호스트**: 물리적인 서버를 통째로 빌리는것, 가장 비쌈

### 추가 기능
- **VMSS (VM Scale Sets)**: AWS Auto Scaling Group과 동일한 기능
    - 메트릭 기반: VM 그룹의 평균 cpu 사용량 등에 따라 스케일 인/아웃
    - 일정 기반: 일정 별로 스케일 인/아웃
- **Azure Bastion**: VM에 공용 IP를 할당하지 않고 사설 IP를 이용하여 Azure 포털에서 접속할 수 있도록 해주는 PaaS 서비스
- **JIT (Just-In-Time) VM 액세스**: 평소에는 관리 포트(ssh, rdp)를 닫아두었다가 관리자가 요청할때만 특정 시간 동안 특정 IP에 대해 포트 오픈
- **Azure Disk Encryption (ADE)**: Bitlocker나 DM-Crypt를 사용하여 VM의 OS및 데이터 디스크 자체 암호화

## App Service
> 웹 애플리케이션, REST API, 모바일 백엔드를 호스팅하기 위한 HTTP기반의 PaaS서비스로 인프라 관리를 Azure에 맡기고 개발자는 코드에만 집중

### 기본 구조
- **App Service Plan**: 하드웨어 틀
    - 웹 앱이 실행될 물리적인 컴퓨팅 리소스의 묶음 (CPU, 램, 스토리지)
    - 과금의 기준
- **App Service**: 소프트웨어
    - 실제 개발자가 배포한 코드 단위
    - 하나의 ASP 위에 여러개의 App Service를 올려서 자원 공유 가능

### 보안 및 네트워크
- **Private Endpoint**: 외부 인터넷을 통한 접근을 차단하고 Vnet 내부의 사설 IP로만 접속 가능하도록 구성
- **Vnet Integration**: App Service가 Vnet 내부에 있는 프라이빗 리소스와 사설 IP로 통신할 수 있도록 해줌
- **SSL/TLS**: 플랫폼 자체에서 인증서를 관리해주며 무료 관리형 인증서 제공 (App Service Managed Certificate)

### 무중단 운영 및 확장성
- **Deployment Slots**: 프로덕션 환경과 똑같은 복제본(스테이징)을 만듬
    - 새 버전을 스테이징 환경에 배포해 테스트 한 뒤 교환을 통해 실시간으로 변경 가능
- **스케일링**:
    - 스케일업: ASP의 하드웨어 등급을 올림
    - 스케일아웃: 동일한 스펙의 서버 대수를 늘림, 오토스케일링 지원


## Azure Functions
> 인프라 관리 없이 코드 단위로만 실행되는 이벤트 기반 서버리스 컴퓨팅 서비스 (=AWS Lambda)

### 주요 특징
- **인프라 X**: OS 업데이트, 스케일링, 로드밸런싱을 모두 Azure에서 알아서 처리, 개발자는 코드 작성에만 집중
- **이벤트 기반**: 함수는 평소에는 정지해 있고 특정 이벤트가 발생할 때만 코드 실행
- **마이크로 과금**: 서버를 켜둔 시간이 아닌 함수가 실행된 횟수와 걸린 시간 기반으로 과금

### 트리거와 바인딩
- **트리거**: 누가 이 함수를 깨울 것인가 (함수당 1개 설정)
    - ex) 사용자가 API를 호출할 때, 매일 자정일때, 저장소에 사진이 업로드 됐을때 등
- **바인딩**: 어떤 데이터와 연결할 것인가
    - 입력 바인딩: 함수가 시작될때 자동으로 데이터를 가져옴
    - 출력 바인딩: 함수가 끝날때 데이터를 자동으로 저장하거나 전송

### 호스팅 및 요금제
- **소비 플랜(Consumption Plan)**
    - 진정한 서버리스, 0 에서 수천까지 자동 확장되며 쓴만큼 지불
    - 단점: 콜드 스타트, Vnet 접속 불가
